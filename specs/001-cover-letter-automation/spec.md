# Feature Specification: 자소서 작성 자동화 서비스

**Feature Branch**: `001-cover-letter-automation`
**Created**: 2026-04-07
**Status**: Draft
**Input**: 자소서 작성이 빈번한 취준생을 위한 서비스 — 기업 조사, 직무 분석, 매핑 테이블 생성, 답변 작성을 자동화

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사용자 프로필 등록 (Priority: P1)

취준생이 처음 서비스를 사용할 때 자신의 포트폴리오(프로젝트 경험, 직무 역량, 수상 내역 등)와 합격 자소서 파일을 업로드한다. 시스템은 이를 분석해 지원자의 경험·역량·문체·어조 프로필을 DB에 저장한다. 이후 모든 자소서 작성 세션에서 이 프로필을 재사용한다.

**Why this priority**: 이후 모든 기능의 전제 조건이다. 사용자 프로필 없이는 맞춤형 답변 생성이 불가능하다.

**Independent Test**: 사용자가 파일을 업로드하고 분석 완료 후, 저장된 프로필 요약(경험 목록, 핵심 역량, 문체 특성)을 화면에서 확인할 수 있으면 독립적으로 가치를 제공한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 처음 접속한 상태, **When** TXT 파일을 업로드하거나 포트폴리오·합격 자소서 텍스트를 직접 붙여넣기하면, **Then** 시스템이 경험 항목, 역량 키워드, 문체 특성을 추출하여 요약을 표시한다
2. **Given** 프로필 추출이 완료된 상태, **When** 문체 프로필 요약이 표시되면, **Then** 사용자가 항목을 수정·삭제·추가한 뒤 확인 버튼으로 저장할 수 있으며, 저장 전까지는 DB에 반영되지 않는다
3. **Given** 프로필이 이미 저장된 상태, **When** 사용자가 추가 텍스트를 입력하거나 새 TXT 파일을 업로드하면, **Then** 기존 프로필에 내용이 병합되어 업데이트되며 동일하게 검토·확인 단계를 거친다
4. **Given** 텍스트 입력창이 비어있거나 TXT 파일이 빈 파일인 경우, **When** 사용자가 분석을 시작하면, **Then** 입력 내용이 없음을 안내하는 메시지를 표시하고 입력을 유도한다

---

### User Story 2 - 기업·직무 분석 및 문항 분석 (Priority: P2)

사용자가 지원 기업명과 지원 직무를 입력하면, 시스템이 웹 검색과 기존 DB를 조합해 기업 정보(주요 제품·서비스, 인재상, 최근 뉴스), 업계 동향, 직무 페인 포인트를 수집·분석한다. 이후 자소서 문항을 입력하면 해당 문항이 어떤 역량을 확인하려는지 분석 결과를 제공한다.

**Why this priority**: 문항 분석 없이는 맞춤형 답변 방향 설정이 불가능하다. 기업 분석 결과는 반복 지원 시 재사용되어 반복 작업을 없애준다.

**Independent Test**: 기업명 + 직무 입력 후 분석 요약(기업 특징, 직무 페인 포인트, 문항 분석)이 화면에 표시되고, 사용자가 내용을 수정·보완할 수 있으면 독립적으로 가치를 제공한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 "삼성전자 + 소프트웨어 개발" 입력, **When** 분석 실행, **Then** 기업 개요, 업계 동향 요약, 직무 주요 업무·페인 포인트가 표시된다
2. **Given** 동일 기업에 두 번째 지원 시, **When** 기업 분석 요청, **Then** 이전 분석 결과를 DB에서 불러오고 최신 뉴스만 추가로 검색하여 연산 비용을 절감한다
3. **Given** 기업 분석 완료 후 자소서 문항 입력 시, **When** 문항 분석 실행, **Then** "이 문항이 확인하려는 역량"과 "신입 지원자에게 기대하는 수준"이 명시된다
4. **Given** 분석 결과가 표시된 상태, **When** 사용자가 특정 항목을 수정하면, **Then** 수정 내용이 저장되고 이후 단계에 반영된다

---

### User Story 3 - 경험-문항 매핑 테이블 생성 (Priority: P3)

문항 분석 결과와 사용자 프로필을 기반으로, 해당 문항 답변에 활용 가능한 사용자의 경험 목록과 각 경험이 문항 의도와 어떻게 연결되는지를 매핑 테이블로 제시한다. 사용자는 어떤 경험을 중심으로 답변을 작성할지 선택·수정한다.

**Why this priority**: "어떤 경험을 쓸지" 결정이 답변 품질의 핵심이며, 이 단계에서 사용자가 방향을 확정해야 최종 답변 생성이 의미 있다.

**Independent Test**: 매핑 테이블이 표시되고 사용자가 경험을 선택·수정한 뒤 저장하면 독립적으로 가치를 제공한다.

**Acceptance Scenarios**:

1. **Given** 문항 분석 완료 상태, **When** 매핑 테이블 생성 요청, **Then** 사용자 경험 목록과 각 경험의 적합도 점수, 연결 근거가 표로 표시된다
2. **Given** 매핑 테이블이 표시된 상태, **When** 사용자가 경험을 추가·제거·수정하면, **Then** 변경 내용이 저장되고 답변 생성 단계에 전달된다
3. **Given** 사용자 프로필에 관련 경험이 없는 경우, **When** 매핑 테이블 생성 시, **Then** "관련 경험 없음"으로 표시하고 직접 입력 옵션을 제공한다
4. **Given** 동일 경험이 복수 문항에 배정된 경우, **When** 매핑 테이블 확정 시, **Then** 시스템이 각 문항에서의 활용 방식(역할·깊이·전개 수준)이 서로 다른지 검증하고, 동일한 방식으로 중복 사용될 경우 경고를 표시한다
5. **Given** 동일 경험이 복수 문항에 배정되었지만 활용 방식이 다른 경우(예: A문항에서 배경 언급 / B문항에서 구체적 전개), **When** 매핑 확정 시, **Then** 경고 없이 정상 진행된다

---

### User Story 4 - 자소서 답변 생성 및 품질 검증 (Priority: P4)

선택된 경험·매핑 결과를 바탕으로 자소서 문항 답변 초안을 생성한다. 시스템은 생성 직후 자동으로 답변 품질을 자가진단하여 AI 특유 표현·어색한 전개·문맥 단절 등의 문제 목록을 표시한다. 사용자가 승인하면 해당 문제를 반영해 재작성하며, 이후에도 수정 지시를 반복하여 최종 답변을 완성한다.

**Why this priority**: 최종 산출물이지만 앞선 3단계가 완료되어야 의미 있는 결과물이 나온다.

**Independent Test**: 매핑 테이블 선택 완료 후 답변 초안이 생성되고, 사용자가 수정 요청을 통해 재생성할 수 있으면 독립적으로 가치를 제공한다.

**Acceptance Scenarios**:

1. **Given** 매핑 테이블 확정 상태, **When** 답변 생성 요청, **Then** 선택된 경험과 기업 분석을 반영한 답변 초안이 표시된다
2. **Given** 답변 초안이 생성된 직후, **When** 자가진단이 자동 실행되면, **Then** AI 특유 표현·문맥 단절·어색한 전개 등 문제 항목 목록이 답변 옆에 표시된다
3. **Given** 자가진단 문제 목록이 표시된 상태, **When** 사용자가 재작성을 승인하면, **Then** 문제 목록을 반영한 수정 답변이 재생성된다. 사용자가 무시하면 현재 답변을 그대로 유지한다
4. **Given** 답변 초안이 표시된 상태, **When** 사용자가 "더 구체적으로", "분량 줄여줘" 등 수정 지시, **Then** 지시에 맞게 수정된 답변이 재생성된다
5. **Given** 글자 수 제한이 설정된 문항에서 답변이 생성된 경우, **When** 생성된 답변이 제한을 초과하거나 90% 미만인 경우, **Then** 시스템이 자동으로 재작성을 시도하며(최대 3회), 3회 이후에도 범위를 벗어나면 현재 결과와 함께 사용자에게 알림을 표시한다
6. **Given** 사용자가 최종 답변 확정 시, **When** 저장 요청, **Then** 해당 기업-직무-문항-답변 조합이 DB에 저장되어 이후 참고 자료로 활용 가능하다

---

### Edge Cases

- 동일 기업에 여러 직무로 동시 지원 시: 기업 분석은 공유하되 직무 분석은 별도로 관리한다
- 웹 검색 결과가 없거나 오래된 정보만 있을 때: 수집 가능한 범위 내 결과를 제공하고 사용자에게 정보 신선도를 표시한다
- 문항 글자 수 제한이 있을 때: 사용자가 글자 수 제한을 입력하면 답변 생성 시 자동 반영하며, 90~95% 범위를 목표로 한다
- 글자 수 자동 재작성 3회 후에도 범위 미달/초과 시: 현재 결과를 사용자에게 표시하고 수동 수정 지시를 안내한다
- 동일 경험이 모든 문항에 동일한 방식으로 배정될 때: 매핑 테이블 확정 단계에서 경고를 표시하고 재배정을 권고한다
- 자가진단 결과 문제가 없을 때: "품질 이상 없음" 메시지를 표시하고 사용자가 다음 단계로 바로 진행할 수 있도록 한다
- 자가진단에서 동일 문제가 재작성 후에도 반복될 때: 사용자에게 알리고 수동 수정을 안내한다
- 사용자가 피드백 없이 이전 단계로 돌아가려 할 때: 현재 단계 진행 내용을 임시 저장한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자가 입력한 텍스트(직접 붙여넣기) 또는 업로드한 TXT 파일에서 텍스트를 읽어 분석해야 한다. PDF·DOCX 파싱은 MVP 범위 외이다
- **FR-002**: 시스템은 파일 분석 후 추출된 사용자 프로필(경험, 역량, 문체)을 사용자가 검토·수정할 수 있는 형태로 표시해야 하며, 사용자가 확인한 이후에만 DB에 저장하고 이후 세션에서 재사용해야 한다
- **FR-002a**: 시스템은 답변 생성 시 저장된 사용자 문체 프로필(문장 길이·어조·표현 방식)을 생성 조건으로 반영하여 초안을 생성해야 한다. 문체 일치 검사는 자가진단 단계가 아닌 생성 단계(프롬프트 수준)에서 처리한다
- **FR-003**: 시스템은 기업명과 직무를 입력받아 다음 3개 소스에서 관련 정보를 수집·통합해야 한다
- **FR-003a** (DART 사업보고서): 금융감독원 DART API를 통해 최근 3개년 사업보고서에서 지원 직무 관련 '주요 제품 및 서비스, 지식재산권, 시장현황, 주요계약 및 연구개발활동' 항목을 수집해야 한다. `DART_API_KEY` 미설정 시 해당 소스는 건너뛰고 나머지 소스로 진행한다
- **FR-003b** (뉴스 수집): Naver News API를 통해 직무 관련 최신 이슈를 수집해야 한다. 수집된 기사가 5건 미만인 경우 Firecrawl 등의 크롤링 API로 대체할 수 있다
- **FR-003c** (공식 홈페이지): 기업 공식 홈페이지에서 인재상, 비전·미션 등 채용 관련 정보를 수집해야 한다. 홈페이지 구조에 따라 수집이 불가능한 경우 사용자에게 안내해야 한다
- **FR-004**: 시스템은 동일 기업의 분석 결과를 DB에 캐싱하여 재조회 시 재사용해야 한다
- **FR-005**: 시스템은 자소서 문항을 분석하여 해당 문항이 측정하려는 역량을 제시해야 한다
- **FR-006**: 시스템은 사용자 경험과 문항 의도를 연결하는 매핑 테이블을 생성해야 한다
- **FR-006a**: 매핑 테이블 확정 시, 동일 경험이 복수 문항에 동일한 활용 방식(역할·깊이·전개 수준)으로 중복 배정될 경우 경고를 표시해야 한다. 활용 방식이 서로 다르다면 중복 배정이 허용된다
- **FR-007**: 각 분석 단계(기업 분석, 문항 분석, 매핑 테이블, 답변 초안) 완료 후 사용자가 결과를 검토·수정할 수 있어야 한다
- **FR-008**: 시스템은 확정된 매핑 기반으로 자소서 답변 초안을 생성해야 한다 *(LLM: Gemini Pro — 중비용)*
- **FR-009**: 사용자는 수정 지시를 통해 답변을 반복 재생성할 수 있어야 한다
- **FR-009a**: 글자 수 제한이 설정된 경우, 시스템은 답변 생성 후 글자 수가 제한의 90~95% 범위에 들어올 때까지 최대 3회 자동으로 재작성해야 한다. 3회 이후에도 범위를 벗어나면 현재 결과와 함께 사용자에게 알림을 제공해야 한다
- **FR-010**: 완성된 답변은 기업-직무-문항 조합으로 저장되어 이후 참조 가능해야 한다
- **FR-011**: 시스템은 답변 생성 직후 자동으로 자가진단을 실행하여 AI 특유 표현, 문맥 단절, 어색한 전개 등의 문제 항목을 탐지하고 목록으로 표시해야 한다. 문체 일치 여부는 자가진단 범위에 포함하지 않는다 (생성 단계에서 처리) *(LLM: Gemini Pro Thinking — 고비용)*
- **FR-011a**: 자가진단 결과 표시 후, 사용자가 재작성을 승인한 경우에만 문제 반영 재작성을 수행해야 한다. 사용자가 무시·스킵하면 현재 답변을 그대로 유지해야 한다
- **FR-012** *(Future)*: 최종 완성된 답변에 대해 외부 AI 탐지 API 또는 오픈소스 모델을 활용한 AI 생성 여부 검증 단계를 제공한다. MVP 범위 외이며 추후 확장 지점으로 예약한다

### Key Entities

- **UserProfile**: 사용자의 경험 목록, 역량 키워드, 문체·어조 특성(문장 길이·표현 방식·선호 어조), 출처 파일 참조. 사용자가 검토·수정 후 확인한 버전만 저장됨
- **CompanyAnalysis**: 기업명, 분석 일자, 주요 사업·서비스, 인재상, 업계 동향, 경쟁사 비교, 원본 수집 소스
- **JobAnalysis**: 지원 직무명, 연결된 기업 분석 참조, 직무 주요 업무, 페인 포인트, 기대 역량
- **Question**: 문항 텍스트, 글자 수 제한, 목표 글자 수 범위(90~95%), 문항 분석 결과(측정 역량), 연결된 직무 분석 참조
- **MappingTable**: 문항 참조, 선택된 경험 목록, 각 경험의 연결 근거·적합도·활용 방식(역할/깊이/전개 수준)
- **CoverLetterDraft**: 문항 참조, 매핑 테이블 참조, 생성된 답변 텍스트, 버전 이력

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 기업 분석부터 첫 답변 초안 생성까지 사용자 능동 작업 시간이 기존 수작업 대비 70% 이상 감소한다
- **SC-002**: 동일 기업 재지원 시 기업 분석 단계를 생략하고 캐시를 활용하여 해당 단계 소요 시간이 0분에 가깝다
- **SC-003**: 각 단계 완료 후 사용자가 1회 이상의 피드백 수정 없이 다음 단계로 넘어갈 수 있다 (강제 수정 없음)
- **SC-004**: 생성된 답변 초안은 사용자가 3회 이내의 수정 지시로 최종본 수준에 도달한다
- **SC-005**: 동일 기업 + 직무 조합에 대해 여러 문항을 처리할 때 기업·직무 분석 결과가 일관되게 적용된다

## Clarifications

### Session 2026-04-07

- Q: 동일 경험 중복 배정 금지 규칙의 엄격도는? → A: 완전 금지 아님. 동일 경험이라도 문항마다 활용 방식(역할·깊이·전개 수준)이 다르면 허용. 동일 방식으로 중복 배정 시 경고 표시
- Q: 글자 수 초과 시 자동 재작성의 범위는? → A: 글자 수 범위(90~95%) 충족까지 최대 3회 자동 재작성, 3회 이후에도 범위 미달/초과 시 사용자에게 알림
- Q: AI 자가진단 단계의 동작 방식은? → A: 답변 생성 후 자동 진단 실행, 문제 목록 표시 후 사용자 승인 시에만 재작성. 사용자가 무시하면 현재 답변 유지
- Q: 사용자 문체 프로필을 DB 저장 전 검토·수정할 수 있어야 하는가? → A: 추출 후 프로필 요약을 사용자가 검토·수정 가능, 확인 버튼으로 저장. 확인 전에는 DB 비반영
- Q: AI 자가진단의 평가 범위에 문체 일치 검사도 포함하는가? → A: 아니오. 자가진단은 AI 특유 표현·어색한 흐름만 검사. 문체 일치는 생성 단계(프롬프트)에서 반영하며 진단 범위 외

## Assumptions

- MVP 단계에서는 단일 사용자(본인)를 가정하며 별도 회원 인증 시스템 없이 로컬 단일 사용자로 운영한다
- 기업 정보 수집은 DART API(사업보고서), Naver News API(뉴스·이슈), 공식 홈페이지(인재상·비전) 3개 소스를 병렬 수집하여 통합한다. `DART_API_KEY` 미설정 시 해당 소스 스킵. Firecrawl은 Naver News 결과 불충분 시 대체 옵션
- 사용자 정보 입력은 텍스트 직접 붙여넣기 또는 TXT 파일 업로드로 제한한다. PDF·DOCX 파싱은 MVP 범위 외이다
- 처음의 배포 환경은 로컬 Docker Compose이며, 이후 클라우드 서빙으로 확장한다
- 답변의 언어는 한국어로 고정한다 (영문 자소서는 MVP 범위 외)
- 각 분석 단계 결과는 다음 단계 진행 전 사용자 확인을 필수로 거친다
