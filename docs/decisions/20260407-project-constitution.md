# 프로젝트 헌법 및 개발 원칙 수립

**날짜**: 2026-04-07
**상태**: 승인

## 배경

TrendOps는 AI 기반 뉴스 요약 서비스이면서, MLOps 엔지니어 취업을 위한 포트폴리오 프로젝트이다.
프로젝트가 진행되면서 코드 결과물만으로는 의사결정 역량을 증명하기 어렵고, 일관된 개발 방향이 없으면
기술 부채가 쌓이거나 확장할 때 전면 재작성이 필요해질 수 있다는 문제를 인식했다.

또한 다음 니즈가 있었다:
- 빠른 로컬 MVP → 이후 클라우드 확장의 단계적 개발 로드맵 필요
- 프론트부터 인프라까지 일관된 기술 스택과 폴더 구조 정의 필요
- 코드 품질 기준과 CI/CD 게이트 명확화 필요

## 결정

`.specify/memory/constitution.md`에 TrendOps Constitution v1.0.0을 수립한다.
5개의 핵심 원칙과 기술 스택·프로젝트 구조·개발 워크플로우를 정의한다.

**핵심 원칙 요약:**

1. **MLOps 중심 설계** — 모든 기술 결정은 MLOps 파이프라인 경험 축적을 목표로 한다
2. **컨테이너 우선** — 모든 서비스는 Docker 컨테이너로 관리한다
3. **MVP 우선, 확장 가능 설계** — 로컬 MVP → 클라우드 확장의 단계적 접근
4. **코드 최소화 & 품질** — 최소 코드 + pre-commit 훅 + 기능 완료 후 코드리뷰 필수
5. **의사결정 문서화** — ADR 형식으로 모든 주요 기술 결정을 `docs/decisions/`에 기록

## 대안 검토

| 대안 | 장점 | 단점 | 기각 이유 |
|------|------|------|----------|
| 별도 원칙 없이 개발 진행 | 초기 속도 빠름 | 방향성 없이 기술 부채 축적 | 포트폴리오 품질과 확장성 위해 기각 |
| 영어로만 문서 작성 | 국제적 가독성 | 본인 사고를 한글로 정리하는 것이 더 명확 | 한글 문서가 실무 의사결정 과정을 더 잘 반영 |
| ADR 없이 README에 통합 | 파일 수 감소 | 결정 추적이 어렵고 히스토리 관리 불가 | 결정 이력 추적을 위해 별도 파일 유지 |

## 결과 및 트레이드오프

**얻는 것:**
- 일관된 개발 방향성 — 기능 추가 시 원칙 기반으로 판단 가능
- 의사결정 이력 — 포트폴리오에서 "왜" 를 설명할 수 있는 근거
- 확장성 보장 — 로컬 MVP에서 클라우드로 전환 시 재작성 최소화

**감수하는 것:**
- ADR 작성 오버헤드 — 작은 변경에도 문서 작성 부담
- 원칙 준수 확인 비용 — PR 리뷰 시 Constitution Check 게이트 추가

**향후 재검토 조건:**
- 팀원이 합류할 경우 원칙 적용 범위와 예외 케이스 재논의
- 클라우드 전환 단계(Phase 3+)에서 K8s 관련 원칙 항목 보강 필요
